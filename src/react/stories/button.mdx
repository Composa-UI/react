# Button

A button lets someone initiate an action, acknowledge something, make a choice, or navigate.

<DocTabs>

<DocTab label="Usage">

<Canvas of={ButtonStories.Button} />

### Overview

Buttons are 24px tall by default, come in two core styles — **primary** and **secondary** — and
can optionally include a leading icon. A larger 32px size is available for prominent,
transactional moments, and buttons can stretch to fill the width of a sidebar.

### When to use

- **Primary button** — the single most important action in a view, like "Share". In a dialog
  used for navigation, prefer terms like "Next", "Done", and "Got it".
- **Secondary button** — for an action that isn't critical, or one paired with a primary button
  (such as "Skip" or "Cancel"); it uses the outline style.
- **Destructive button** — when an action is dangerous (like "Delete"), use the red fill to
  signal that the user needs to be careful. If the destructive action is the *secondary* one
  (sitting next to a primary button), fall back to the secondary destructive outline style.
- **Link button** — when a button should visually read as a link: no background, brand-colored
  text.
- **Ghost button** — when you want link-like appearance but real button behavior and focus
  states for assistive tech. It looks like a link but functions like a button.

### When not to use

- **Don't** use a button for a setting that takes effect immediately — that's a Switch.
- **Don't** stack multiple primary buttons in one group; there is one primary action per view.
- **Don't** use a destructive fill for a non-dangerous action; reserve red for genuinely risky
  or irreversible operations.

### Variants

The button family shares one anatomy and state model; the variants differ in size, width, and
interaction model.

<Canvas of={ButtonStories.IconButton} />

- **Icon button** — an icon-only button in a 24×24 container that guarantees a consistent click
  target. In most cases, pair it with a tooltip so its purpose is clear. The common pattern is a
  light grey hover state and a blue filled active state when a related panel is open.

<Canvas of={ButtonStories.Toggle} />

- **Toggle (icon) button** — an icon button with a toggleable on/off state. Clicking flips
  between the two states. The "on" state recolors the glyph to selection-blue on a pale-blue
  tint, the visible signal that the toggle is active. A dialog-toggle variant uses a selected
  blue fill for "on"; a stroked variant adds a hairline border to ground the icon on larger
  surfaces.

<Canvas of={ButtonStories.SplitButton} />

- **Split button** — a primary action paired with an adjacent chevron half that opens related
  choices, separated by a 1px divider, so the main action and its alternates live in one control.

Beyond these, the same button supports a **Large** size (32px tall, primary or secondary, with
an optional icon) for prominent transactional actions, a **Wide sidebar** layout that fills the
width of a sidebar with 16px margins on each side (available as primary, secondary, and ghost),
and an **Inverse** style for the "on" state of a button that flips between two labels — for
example "Install" / "Installed". The inverse style is paired with a secondary button for the
"off" state.

### Anatomy

<Canvas of={ButtonStories.Anatomy} />

- **Container** — a rounded rectangle (radius `radius-medium`) holding the label and optional
  icon. The default container is 24px tall; the large size is 32px tall.
- **Label** — the action text, set in `body.medium`.
- **Leading icon** (optional) — a 24px glyph before the label (icon + text variant).
- **Enter glyph** (optional) — primary and link buttons may carry a trailing `⏎` enter glyph in
  `color.text.onbrand-secondary` to signal the default/confirm action.
- **Icon button** — icon-only, with a 24×24 click-target container and no label slot.
- **Split button** — an action half + a 1px divider + a chevron menu half.

### Content / label rules

- Label buttons with a clear, action-first verb naming what happens ("Share", "Delete", "Next").
- For navigation in a dialog, use "Next", "Done", or "Got it".
- For inverse on/off pairs, use the two states as the label ("Install" / "Installed").
- Icon buttons have no visible label, so they should almost always carry a tooltip.

### Placement & order _(proposed)_

- In a dialog footer or action bar, place the **primary on the right**, with the secondary
  (or "Cancel") to its left, separated by `--composa-space-2`.
- When the pair is **stacked vertically** (such as a narrow sidebar), the **primary sits on
  top**, with the secondary/Cancel below it.

### Behaviors

- **Primary** carries the most visual weight; **secondary** is the quieter outline. The primary
  fill automatically shifts hue to match the current product theme — a primary button is not
  always one fixed color.
- **Active / pressed** darkens the fill (or, for outline styles, adds a pressed background under
  the border).
- **Focus** shows a `color.border.selected` outline; filled buttons add an inset 2px ring so the
  focus indicator reads against the fill.
- **Disabled** dims the control to the disabled fill and text tokens.
- **Icon buttons** add a light grey hover fill (`color.bg.hover-transparent`) and a pointer
  cursor; a related-panel-open state can use a blue filled active treatment.
- **Toggle buttons** flip between on and off on click; the on state carries a grey or selected
  fill with a recolored glyph. A single-choice toggle does not deselect itself on a repeated
  activation.

| State | Primary | Secondary | Destructive | Link | Ghost |

| Active / pressed | `color.bg.brand-pressed` | `color.bg.pressed` under border | `color.bg.danger-pressed` | `color.bg.brand-tertiary` | `color.bg-transparent.secondary-pressed` |
| Focus | `color.border.selected` outline + inset 2px ring over `color.bg.brand` | `color.border.selected` | `color.border.selected` over `color.bg.danger` | `color.border.selected` | `color.border.selected` |

While an action is pending the button enters a **loading** state: it keeps its current size (the Spinner occupies the 24×24 leading-icon slot so width doesn't jump) and holds the rest fill rather than dimming, so the action still reads as the one in progress. The control is non-interactive — disabled for input/hit-testing — for the duration. _(proposed)_

#### Screen reader & labeling

- Every member is a native `<button>` exposing the `button` role; its visible label is the
  accessible name.
- **Icon buttons have no visible text**, so always provide an accessible label (and usually a
  tooltip) — otherwise the control is unlabeled to assistive tech.
- Ghost buttons look like links but are real buttons, so they expose button semantics and focus
  states for screen readers.
- A toggle button announces its state with `aria-pressed`, so assistive tech reports on vs. off;
  the recolored glyph is a visible reinforcement, not the only signal.
- A split button exposes two separately focusable, separately labeled controls — the action half
  and the menu half (named `${label} options` by default).
- A disabled button is announced as unavailable and is removed from the activation flow.
- Don't rely on color alone — destructive intent is carried by the label verb ("Delete…") and
  placement, not only the red fill.

#### Focus

- The focused button shows a `color.border.selected` outline; filled buttons add an inset 2px
  ring so the focus indicator stays visible against the fill, meeting the visible-focus
  requirement.
