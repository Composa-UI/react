# Comments

Comments let people leave threaded feedback on a surface — pinning a remark to a spot, reading and replying in a thread, or composing a fresh note on the canvas or in the right-hand sidebar.

<DocTabs>

<DocTab label="Usage">

<Canvas of={CommentsStories.Playground} />

### Overview

A comment is a threaded conversation anchored to a place on the surface. Clicking a comment pin opens a **thread window** where the conversation can be read and replied to. New comments can be composed directly on the canvas, and the full set of comments (and similar content types) is listed in a **dismissible sidebar** on the right when the comment tool is selected.

### When to use

- **Comments** — to leave feedback or start a conversation tied to a specific spot on a surface, where the discussion lives alongside the work rather than in a separate tool.
- The sidebar surfaces every comment at once; the thread window focuses one conversation; the on-canvas composer starts a new one in place.

### Variants / parts

The thread window opens from a pin and holds the title bar, the replies, and the reply input.

#### Sidebar rows

Each comment appears as a row in the right-hand sidebar. The row shows an avatar list, the author name, a page reference, a timestamp, a truncated message preview, and a reply count. The unread dot sits on the **right** of the inner card.

<Canvas of={CommentsStories.SidebarRow} />

Rows have three interaction states: default (no fill), hover (`color.bg.hover`), and selected (`color.bg.selected`).

<Canvas of={CommentsStories.SidebarRowStates} />

#### Canvas pins

Comment pins anchor conversations to the canvas. An unread pin is filled `color.accent-blue`; a read pin switches to a bordered white circle. Clusters group 2–5 comments into a pill.

### Anatomy

**Thread window** — a 360px floating card on `color.bg`, radius `radius-large`, with a soft shadow:

1. **Title bar** — a 40px bar with the "Comment" title and a right-aligned action group (`icon.24.more`, `icon.24.resolve`, `icon.24.close.small`), separated from the thread by a 1px `color.border`.
2. **Thread** — a column of comment rows, each with an avatar, the author name (`body.large.strong`), a timestamp (`body.large`), and the message body (`body.large`).
3. **Reply input** — the chat input at the foot of the window.

**Comment row** content reads as: avatar, name, "6 hours ago", then the body, for example — _"I love where this is headed, but I'm not quite sure about the spacing here. I think things could be nudged over a bit to get to a tighter rhythm."_

**Sidebar row** — a row in the comment list, with an avatar list, the author name, the page reference and timestamp, a one-line message preview, and a reply count. The unread dot sits on the **right** (rather than the left), which keeps it aligned with the section header in the collapsed state.

**Canvas pin** — a 28×28 circle with a teardrop tail, filled `color.accent-blue` (unread) or `color.bg` with a border (read). Clusters are pill-shaped and hold 1–3 inline avatars.

### Content rules

- **Reply input** placeholder reads "Reply"; the on-canvas composer placeholder reads "Add a comment".
- **Sidebar row** shows the author name, a page reference (for example "#3 · Page 1"), a timestamp ("Just now"), a truncated one-line message preview, and a reply count ("2 replies"). The preview message truncates with a trailing ellipsis, for example — _"What happens if we adjust this to handle a light and dark mode? I'm not sure we're ready to handle that quite…"_
- The reply count reads in secondary text when the thread is read, and switches to brand text when it is unread.

### Behaviors

- **Thread window opens from a pin.** Clicking a comment pin shows the thread view, sized to a clean width for reading and replying.
- **Reply input grows on typing.** As the user starts typing into the reply input, the reply area grows taller to reveal additional controls — **add emoji** (`icon.24.emoji`), **add mention** (`icon.24.mention`), **add image** (`icon.24.image.small`) — and the **send** arrow (`icon.24.send`) flips to its primary state.
- **On-canvas composer** uses the same reply interaction, expanding from a single line (placeholder "Add a comment") to the taller composer as typing begins. It differs from the thread reply only in its floating "outer" radius and shadow, versus the thread reply's inset secondary-background field.
- **Sidebar row hover / selection.** A row has hover and selected styles: hover fills the inner card with `color.bg.hover` and reveals a top-right action group (`icon.24.more`, `icon.24.resolve`); selected fills it with `color.bg.selected`.
- **Sidebar is dismissible.** The comment sidebar appears on the right when the comment tool is selected and can be dismissed.
- **Pin states.** An unread pin is accent-blue; once the thread is read it becomes a bordered white circle. Clicking a pin opens the thread and flips the pin to selected (outline ring + scale). Hovering scales the pin 1.1×.

#### Keyboard

| Key | Result |
|---|---|
| `Tab` | Moves focus into the thread window, then through its title-bar actions, the thread, and the reply input |
| `Enter` | Sends the reply when the reply input has focus |
| `Escape` | Closes the thread window / dismisses the sidebar |

#### Screen reader & labeling

- The reply input has an accessible name from its placeholder ("Reply") or composer placeholder ("Add a comment"); pair it with a label so the field is named to assistive tech.
- Title-bar actions (`more`, `resolve`, `close`) carry text labels rather than relying on the glyph alone.
- A sidebar row announces its author, message preview, reply count, and unread state, so the conversation can be navigated without sight.
- Comment pins use `role="button"` and `aria-pressed` to convey selected state.

#### Focus

- Focus moves in reading order through the window: title-bar actions, then each comment row, then the reply input.
- The reply input shows a visible focus indicator, and typing expands it in place without moving focus.
