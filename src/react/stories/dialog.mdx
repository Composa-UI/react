# Dialog

A dialog places content over the page, preventing interaction with the underlying page until the user completes a task or dismisses it.

<DocTabs>

<DocTab label="Usage">

<Canvas of={DialogStories.Playground} />

### Overview

A dialog is a window that floats above the page, paired with a scrim that dims and blocks the content behind it. It holds a self-contained task — renaming a layer, creating a project, confirming a delete, sharing a file — and stays up until the user finishes or dismisses it. The goal is to be as minimally intrusive as possible, so the focus stays on the content inside the dialog.

### When to use

- **Modal dialog** — when the task must be completed or dismissed before the user does anything else. The scrim blocks the page and background content shouldn't be readable while the dialog is up. Use it for focused, blocking tasks: forms, confirmations, sharing.
- **Non-modal surface instead** — for lightweight, non-blocking guidance that sits beside the work rather than over it (tutorial bubbles, feature education popovers). These point at a target and don't dim the page.
- **Match the width to the depth of controls** — text-only dialogs stay narrow; dialogs with labeled inputs, dropdowns, and checkboxes go wider (see Style → Structure).

### Anatomy

<Canvas of={DialogStories.Anatomy} />

A dialog is assembled from three standardized pieces — a header, a body, and a footer — composed on the dialog window. Always use the dialog-specific sub-components rather than rebuilding these by hand.

**Header** — ready-made title-and-controls bar. A left-aligned **title** and a right-aligned group of **icons**: an optional **Copy Link** link, an optional **add/plus** action, and a **close** button. A 1px border sits along the bottom. Variants cover Default, Navigation, Tabs, and Dropdown headers, wired to visibility props so you can toggle pieces from the call site.

<Canvas of={DialogStories.Header} />

**Footer** — ready-made action bar. The right side carries the call-to-action buttons: a primary **Action** button (brand fill) and an optional secondary **Cancel** button (outlined). The left side can show, by prop, a **stepper** ("1 of 3"), **helper text**, a **checkbox** ("Don't show this again"), or an **action link** ("Advanced"). A 1px border sits along the top when enabled. The footer covers a range of cases — action, pagination, and share-modal footers among them.

<Canvas of={DialogStories.Footer} />

**Row** — a body building block for list-style content: header rows, label rows, label-with-description rows, avatar-with-label rows, icon-with-label rows, dropdown rows, and tab rows, each with a hover state. Used to build out sharing lists and property panels inside the body.

<Canvas of={DialogStories.Row} />

**Structured body** — for richer dialogs, the body composes inputs and rows into a full layout (a sharing panel, a create-project form). Body content types include single- and multi-line text inputs, dropdowns, a code-embed field, and invite controls.

#### Anatomy — numbered parts

1. **Scrim** — a full-page backdrop in `color.modalbackdrop` that dims and blocks the page beneath. Clicking it dismisses the dialog.
2. **Window** — the floating surface: `color.bg` fill, `radius-large` corners, and the `elevation-500-modal-window` shadow.
3. **Header** — title plus controls (close, and optional link / add), with a bottom border.
4. **Body** — the content area; inputs, rows, and text. Overflows beneath the footer with a scrollbar when the dialog hits its height limit.
5. **Footer** — actions and optional stepper / helper text / checkbox, with a top border.
6. **Scroll affordance** — a 4 × 70px `radius-full` pill shown when body content exceeds the visible height.

### Content rules

- **Header title** — short and without trailing punctuation (e.g. "Create new project", "Share this team").
- **Body copy** — punctuated, one idea per screen; supporting detail comes after the lead sentence. Example: "We recommend creating workspaces based on product areas, business units, or working groups in your organization."
- **Inputs** — use placeholder hints for examples ("eg Top Secret Ideas"); pair character-limited fields with a counter ("0/160").
- **Footer CTAs** — name the primary action by what it does ("Continue", "Action", "Share"); the secondary is "Cancel".
- **Steppers** read "Step 1 of 3" / "1 of 3" for multi-step flows.

### Behaviors

- **Scrim & blocking** — a dialog renders with a scrim on top of the page's z-index. While it's up, the page underneath can't be interacted with and its content shouldn't be readable.
- **Dismissal** — clicking the scrim or the header **X** dismisses the dialog. (Escape dismisses too — see Accessibility.)
- **Focus trap** — focus stays within the dialog while it's open, so Tab cycles through its controls and not the page behind it.
- **Overflow** — when a dialog reaches its height limit, body content overflows beneath the footer and a scrollbar appears as a visual affordance; the scroll-affordance pill signals there's more to read.

#### States

| State | Treatment |
|---|---|
| Open | Window on `color.bg` over the `color.modalbackdrop` scrim; page behind blocked and dimmed. |
| Scrolled / overflowing | Body scrolls beneath the footer; scroll-affordance pill (`radius-full`) shown. |
| Dismissing | Closes on scrim click, **X**, or Escape; focus returns to the trigger. |

Widths align to the 8px grid: 240 for tutorial bubbles, 320 for text-only dialogs, 480 for dialogs with labeled inputs, dropdowns, checkboxes, and more. The width follows the depth of controls in the body. In special cases (branching, merging, account settings) the dialog sizes relative to the viewport height instead.

```jsx
import { Dialog, OverlayHost } from "@composa-ui/react";

// A dialog floats over the page on a scrim. The scrim, focus management,
// and Escape-to-close are wired at the call site — the Dialog component
// renders the window, header, body, and footer but does not trap focus,
// manage the backdrop, or handle Escape itself. Mount it inside an
// `OverlayHost` ancestor so the window and scrim portal above the page.

<OverlayHost>
  {open && (
    <Dialog
      title="Create new project"
      description="We recommend creating workspaces based on product areas, business units, or working groups."
      primaryLabel="Continue"
      secondaryLabel="Cancel"
      onPrimary={createProject}
      onClose={() => setOpen(false)}
    />
  )}
</OverlayHost>
```

For richer layouts, drop the sub-components (`DialogHeader`, `DialogBody`, `DialogRow`, `DialogFooter`) into `children` instead of using the one-shot `title` / `description` props. An `OverlayHost` ancestor is required so the dialog and its scrim portal above the page — see the **OverlayHost** and **OverlayLayer** overlay primitives.

#### Keyboard

| Key | Result |
|---|---|
| `Tab` | Moves focus to the next control inside the dialog; focus is trapped and does not reach the page behind |
| `Shift` + `Tab` | Moves focus to the previous control inside the dialog |
| `Esc` | Closes the dialog (same as the scrim click or the header X) |
| `Enter` | Activates the focused control / submits the primary action |

#### Screen reader & labeling

- The dialog window exposes a `dialog` role with `aria-modal="true"`, so assistive tech treats the page behind it as inert while the dialog is open.
- Give the dialog an accessible name via `aria-labelledby` pointing at the header title (or `aria-label` when there is no visible title), so the dialog is announced by what it's for.
- The scrim blocks and hides background content, mirroring the visual blocking for assistive tech.

#### Focus

- **Initial focus** moves into the dialog when it opens — to the first interactive control, or the primary action for a confirmation.
- **Focus trap** keeps Tab and Shift+Tab cycling within the dialog while it's open.
- **Return focus** goes back to the element that opened the dialog once it's dismissed (scrim click, X, or Escape).
