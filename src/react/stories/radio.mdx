# Radio

A radio button represents a list of choices where the user can pick **only one option**.
Unlike a list of checkboxes — which toggle independently — radios in a group allow just one
selection at a time.

<DocTabs>

<DocTab label="Usage">

<Canvas of={RadioStories.Playground} />

### When to use

- **Radio** — when the user must choose exactly one option from a mutually exclusive set.
  Selecting one option clears the rest.
- **Checkbox instead** — when more than one option can be selected at once.

### Variants

- **Input** — the standard radio control for option lists: a circular ring with a filled dot
  centered inside when the option is selected.
- **Button** — a selectable pill that reads as a mode switch or segmented choice. The active
  option fills with the brand accent; inactive options show a quiet translucent border.

### Anatomy

<Canvas of={RadioStories.Anatomy} />

- **Control** — a 16 × 16 box rendering the ring, with an 8 × 8 filled dot centered inside
  it when the option is selected (`input` variant only).
- **Label** — optional text naming the distinct option, set to the right of the control.

### Content

- A radio is usually accompanied by a **label** describing a distinct option.
- Keep each label short and specific; it names a single choice within the group.

### Behavior

- **Single selection** — choosing one radio in a group deselects whatever was selected
  before. This is the defining difference from a checkbox.
- **Focus** swaps the control's ring to a selection-blue focus ring; the filled dot, if
  present, stays put.
- The selected and unselected states differ only by the centered dot — the ring is constant.
- **Default selection** — by default, leave no option preselected. If users must be able to
  clear a choice, add an explicit "None" or "Other" option rather than relying on
  deselection — a radio can't be unselected by re-clicking it.

### Dark mode

Both input and button variants resolve all color tokens per theme automatically. No
variant-specific dark treatment is required.

### Do / Don't

- **Do** group radios so the choices are clearly mutually exclusive, and pair each with a
  clear label.
- **Don't** use a radio when more than one option can be selected — that's a Checkbox.

#### Keyboard

The group is a single tab stop (roving focus).

| Key | Result |
|---|---|
| `Tab` | Moves focus into the group — onto the selected radio, or the first radio if none is selected |
| `↑` / `↓` / `←` / `→` | Moves focus and selection between radios within the group |
| `Space` | Selects the focused radio |
| `Tab` (again) | Exits the group, moving focus to the next component |

#### Screen reader & labeling

- The radio exposes a `radio` role with `aria-checked` (`true` / `false`), grouped under a
  `radiogroup` so assistive tech announces the set as mutually exclusive.
- **`label` is the accessible name** — always set a meaningful one, or the radio is
  unlabeled to assistive tech.
- **Group semantics** — wrap the set in a `fieldset` with a `legend` as the group's name,
  and connect each control to its `label` via `for`. The `legend` is the group's accessible
  name in addition to the per-radio labels.
- **Visible group label** — give the group a visible label above the options, in sentence
  case.
- **Helper text** — optional supporting text for the group sits below the label.

#### Focus

- The focused radio swaps its ring to a selection-blue focus ring
  (`color.border.fs-selected-strong`), meeting the visible-focus requirement.
