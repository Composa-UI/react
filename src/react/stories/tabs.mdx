# Tabs

Tabstrips let users navigate between two or more related views. They're a quick way to
switch between related information, with **one tab selected at a time**.

<DocTabs>

<DocTab label="Usage">

<Canvas of={TabsStories.Playground} />

### When to use

- **Tabs** — for sibling content regions in the same view that the user moves between,
  like an inspector's Design / Prototype / Inspect, or page- and panel-level content.
- **SegmentedControl instead** — for an inline mode toggle rather than a content switch.

### Variants

<Canvas of={TabsStories.Variant} />

- **Pill** — the default tabstrip: the selected tab reads as a filled pill (`bg-secondary`,
  `radius-medium`) against unstyled siblings. We sometimes make the first tab an aggregated
  view, such as "All".
- **Underline** — the selected tab carries a colored baseline underline instead of a fill;
  the row anchors the top of the region it controls via a bottom border.

### Anatomy

<Canvas of={TabsStories.Anatomy} />

There are visual cues that help a user quickly distinguish between the current tab and the
other tabs.

- **Current tab** — a 24px-height container styled with `bg-secondary` fill,
  `radius-medium`, `body.medium.strong` type, and `text` color.
- **Other tabs** — a 24px-height container, unstyled, `body.medium`, `text-secondary`.
- **Spacing** — each tab is separated by 4px (`spacer-1`) of space.

### Content

- A tab carries a short **text** label (default "Tab Title").
- A tab may add a **counter badge** to broadcast the difference between newly updated and
  stale information — seen in admin / billing surfaces. The badge sits beside the label
  and reads **Count New** when there's unread content and **Count Inactive** once read.
- **Single tab** — when a panel's tabstrip has only one tab, that tab reverts to
  dialog-title styling: `bg` fill, `text`, `body.medium.strong` — i.e. it drops the
  selected pill.

### Behavior

- **Selection** — exactly one tab is selected at a time; selecting a tab swaps the content
  region below it.
- **Focus** — each tab can be focused individually, and focus can land on the current tab
  or an unselected one, so the user can navigate tabs within a panel by keyboard. A focused
  tab shows a 1px `border.selected` ring without changing the tab's fill or type weight.
- **Overflow** — when tabs overflow, the row scrolls horizontally. Tabs that flow outside
  the parent are masked with gradients that fade into the `bg` background, leaving room for
  a trailing close affordance. _(proposed)_ **Tabs never wrap or stack** — the row always
  scrolls horizontally rather than flowing onto a second line, so the tablist stays a single
  row at any width. _Notes marked (proposed) are inferred, Carbon-benchmarked additions awaiting confirmation._

#### Keyboard

| Key | Result |
|---|---|
| `Tab` | Moves focus into / out of the tabstrip (roving focus — a single tab stop for the whole row) |
| `←` / `→` | Moves focus between tabs within the row |
| `Space` / `Enter` _(proposed)_ | Activates the focused tab and swaps the content region — **manual activation**, so arrow keys move focus without changing the selection until `Space`/`Enter` is pressed |
| `Home` / `End` | Moves focus to the first / last tab |

_Rows/notes marked (proposed) are inferred, Carbon-benchmarked additions awaiting confirmation._

#### Screen reader & labeling

- The tabstrip exposes a `tablist` role; each tab is a `tab` with `aria-selected`
  (`true` / `false`), and the content region is a `tabpanel` associated with its tab.
- **The tab `text` is the accessible name** — keep it short and meaningful so each tab is
  distinguishable to assistive tech.
- A counter badge's number is announced as part of the tab, so unread / stale counts reach
  screen-reader users too.

#### Focus

- Each tab can be focused individually. The focused tab shows a 1px selection-blue ring
  (`color.border.selected`), meeting the visible-focus requirement, whether the tab is the
  current one or an unselected sibling. Focus does not change the tab's fill or type weight.
